    서론

    이 글은 stackoverflow, MSDN 을 참고하여 작성되었습니다.

    이번 글에서는 비트맵 파일 직렬화를 실습한다.
    국내 개발 블로그를 보면 대부분 기본자료형 위주로 포스팅되어 있다.
    너무 기초만 다루는 것 같았다. (내 코드도 기초수준이다)
    네트워크 프로그래밍에서 사용되는 직렬화는 더 공부를 하고 다음에 실습하도록 한다.

    명시적(implicit)으로 직렬화 하는 방법에 대해 알아본다.
    암시적이나 명시적이나 모두 비슷하니
    자세한 설명은 이전 글을 참고하도록 하자.
    이전 글: Document CArchive 기본타입 및 객체 직렬화

    명시적 직렬화도 암시적이랑 비슷하게 CFile 객체를 CArchive 객체가 참조하여 사용한다.

    • CFile 객체를 CArchive 객체가 참조하게 될 때 모드를 정할 수 있다.
    • CArchive::store (저장)
    • CArchive::load (불러오기)

    비트맵의 직렬화

    아래 코드를 설명해보면
    힙 메모리를 할당하여 이미지 객체와 스트림을 만들고(연결) 메모리에 저장한다.
    CreateStreamOnHGlobal() API 함수를 이용하여 메모리에 스트림을 만든다.
    CImage 클래스에서 제공하는 Save() 함수를 이용해 스트림에 데이터를 저장한다.

    데이터가 저장된 힙 메모리에 함수를 이용해 포인터와 메모리 크기를 구하고
    구한 포인터 위치에서 파일의 크기만큼 CArchive 객체를 이용해 직렬화하여 파일을 저장하는 것이다.

    MSDN을 보면 전역(Global) 메모리는 과거에 사용되어졌다고 한다.
    x86 이후의 시스템에서는 16비트 프로그램을 포팅하거나
    16비트 Windows프로그램의 소스코드 호환성 때문에 지원된다고 한다. (도대체 언제적 것인지..)

    GlobalAlloc() 함수는 결국에 디폴트 힙을 할당하는 함수의 래퍼함수로 구현되어 있다고 한다.
    이로 인해 오버헤드가 발생한다는 경고문도 적혀있다.
    x86 환경에선 과거 세그먼트 메모리 구조에서 가상 메모리를 사용 구조로 바뀌었다.

    GMEM_MOVEABLE은 이동이 가능한 메모리를 할당하지만
    메모리는 프로세스 영역 내(힙 영역 내)에서만 움직일 수 있다고 한다.
    반대로 고정된 메모리는 GMEM_FIXED를 이용한다.
    메모리 이동은 힙 메모리 단편화와 연관이 있다.
    특별한 상황이 아니라면 기본적으로 이동이 가능한 메모리를 사용하도록 하자.
    할당 메모리 크기가 0인 이유는 삭제가 가능한 메모리를 할당받기 위해서이다.

    누가 메모리를 왜 삭제하는지는 운영체제에 대한 이해가 필요할 것 같다..
    구글에 아무리 검색해봐도 삭제 가능한(discard) 메모리를 사용하는 이유는 모르겠다.
    일단 0으로 할당함으로 써 스트림에서 자유로운 크기를 만들 수 있다는 것은 확실하다.

    오버헤드가 적은 HeapCreate, HeapAlloc 할당한 함수를 이용해
    스트림을 만들고 저장하면 좋은데 아직 실력이 부족하다..

    GlobalLock() 함수로 메모리의 단편화에서의 이동을 제한하고
    실제 메모리의 첫번쨰 위치를 얻어오는 것이다.

    반드시 리소스를 사용했으면 해제 및 반납하는 작업을 빼먹지 말자.
    이해는 충분히 한 것 같으니 마친다.
