    서론

    MFC에서 제공하는 컨트롤 및 뷰는 가장 단순하고
    가장 기본적인 기능만 제공하고 있다.

    기능을 더 확장 및 변경하기 위해서는
    기본으로 제공하는 컨트롤 및 뷰를 상속받아 기능을 확장해야한다.

    기능을 확장한다는 의미는
    기존 컨트롤에 개발자가 기능을 덧붙여
    새로운 컨트롤을 만든다는 의미와 같다.

    컨트롤 및 뷰를 확장하기 위해서는 서브클래싱을 통해 진행한다.
    서브클래싱을 이용해 윈도우 프로시저에서 보내지는 메시지를 가로채어 기능을 확장한다.
    서브 클래싱은 CWnd 객체에 연결 및 윈도우 프로시저 함수를 대체하는 것이다.

    가로챈 메시지는 윈도우 프로시저에게 전달해 줄 수도 있고
    해당 컨트롤이 처리 후 다른 윈도우 프로시저에게 전달하지 않을 수 있다.

    PreTranslateMessage() 가상함수에서도 메시지를 가로챌 수 있지만
    코드가 길어질 수록 구조가 복잡하게 변할 위험이 있다.

    MFC 2.x, WINAPI에서는 복잡한 함수 호출 과정을 거쳤지만
    4.x 버전이후부터는 과정이 단순하게 개선되었다.

    서브 클래싱은 2가지로 구분한다.

    • 정적 서브 클래싱
    • 동적 서브 클래싱

    정적 서브 클래싱

    정적 서브 클래싱은 간단하게 말해서 DDX와 연결된 서브 클래싱을 의미한다.
    기본 타입(CEdit)으로 연결된 것을 확장타입(CMyEdit)으로 DDX_Control 연결을 하는 것이다.
    가장 중요한건 동적 서브 클래싱이기에 간단히만 알아보도록 한다.

    • DDX_Control(pDX, IDC_MYEDIT, m_edit);

    동적 서브 클래싱

    동적 서브 클래싱을 하기 위해 필요한 함수는
    CWnd::SubclassDlgItem() 및 CWnd::SubSubclassWindow(hWnd) 이다.
    런타임에서 서브클래싱을 할 수 있다.
    프로그램이 실행 되면서 프로시저를 변경할 때 아주 유용하게 쓰인다.
    프로시저가 변경된다는 것은 프로그램의 처리 결과를 바뀌게 된다는 의미이다.

    UnsubclassWindow()

    서브 클래싱을 동적으로 해제해줄 수 있는 함수이다.
    이 함수를 해제하면 기본 윈도우 프로시저가 메시지를 처리하게 된다.
    동적 서브 클래싱이 되지 않은 객체한테 사용하면 Crash를 겪게될 수 있다.

    SubclassDlgItem() 함수

    서브 클래싱할 컨트롤 리소스 ID와 부모 윈도우 포인터를 요구한다. (IDC_MYEDIT)
    부모는 보통 폼에 컨트롤을 배치하기 때문에 View의 포인터를 적어준다.

    A.SubclassDlgItem(B, C);
    -> 부모 C의 배치된 B 컨트롤 프로시저를 A가 지닌 프로시저로 대체한다.

    만약 컨트롤이 단계별로 여러개의 컨트롤로 구성되어 있을 땐
    View가 아닌 해당 컨트롤이 배치되는 부모 컨트롤의 포인터를 적어주도록 한다.

    코드 및 실습

    구현은 다음과 같이 진행한다.

    1. CFormView 를 상속받는 SDI 프로젝트를 만든다. (TESUB)
    2. 폼에 Edit Control(IDC_MYEDIT), Button Control(IDC_BUTTON1) 을 배치한다.
    3. 기존 CEdit을 상속받아 새로운 컨트롤 클래스를 만든다. (CMyEdit)
      • WM_CTLCOLOR 핸들러 추가 및 적용
        • 텍스트 색상: 빨간색
        • 배경모드: 투명
      • WM_CHAR 핸들러 추가 및 적용
        • 숫자만 입력 가능하게 처리
    4. View.h 에서 다음과 같이 진행한다.
      • #include "CMyEdit.h" 추가
      • CMyEdit 변수 선언(m_MyEdit)
      • 플래그 변수 선언(m_initFlag)
    5. View.cpp 에서 IDC_BUTTON1 컨트롤 BN_CLICKED 메시지 처리
      • 버튼을 누르면 서브 클래싱이 되어 프로시저 변경
      • 서브클래싱이 적용된 상태에서 다시 버튼을 누르면 서브 클래싱이 해제
      • 서브클래싱 적용 / 미적용 텍스트 변경 (플래그 값 변경됨)


    주의사항

    WM_CTLCOLOR 핸들러는 새로운 컨트롤의 모습을 갱신하기 위해서는
    서브 클래싱이 적용된 순간 컨트롤을 새롭게 그릴 수 있게 Invalidate() 함수를 호출한다.
    SetFocus() 함수를 이용해서 즉시 화면을 갱신한다.

    서브 클래싱 해제 시에는 서브 클래싱 객체에 접근할 수 없기 때문에
    컨트롤이 배치된 부모 컨트롤(View)에 접근해서 Invalidate() 함수를 호출하여 화면을 새로 그린다.

    결과

    서브 클래싱 버튼을 누르면 프로시저가 대체되면서 숫자만 입력 가능하게 변경된다.
    또 컨트롤 입력창 텍스트의 색상은 빨간색으로 변경된다.

    SubclassWindow() 함수

    프로시저를 바꿀 윈도우의 핸들을 요구한다.
    핸들이 참조하고 있는 객체의 CWnd 객체로 연결되고 윈도우가 가지고 있는 핸들의 프로시저가 변경된다.
    CWnd 객체는 MFC 윈도우 클래스이다.

    이 함수는 WINAPI로 만들어진 윈도우 및 컨트롤을 서브 클래싱할 때 사용된다.
    WINAPI와 혼용해서 사용하도록 하자.

    CWnd 객체 포인터를 가진 객체에게 위 함수를 이용해 서브클래싱을 시도하는 경우 Crash가 발생할 수 있다.
    VERIFY, ASSERT 매크로 함수를 이용해 사용하는 것이 런타임에서 안전하다.
    가능한 WINAPI에서 핸들을 얻어올 때 사용하자.

    WINAPI GetWindow() 함수를 이용해 자식 컨트롤의 핸들을 얻어 사용할 수도 있다.

    A.SubClassWindow(C)
    -> C 핸들을 가지고 있는 윈도우의 프로시저를 A 객체 프로시저로 대체한다.
    -> C의 핸들 값이 A에게 주어진다. (두 핸들이 서로 동일해짐)
    -> CWnd 객체로 연결된다.

    복잡한 실습보다 아주 간단하게 사용만 해보자.

    m_myEdit는 기본 CEdit를 상속받아 만든 CMyEdit 확장 클래스이다.
    서브 클래싱을 하기 전에 m_myEdit은 null이다.
    m_editHwnd 핸들값으로 CWnd 객체를 생성하고 m_myEdit에 연결시킨다.
    객체가 연결됨으로 기본 프로시저가 아닌 CMyEdit 프로시저가 메시지를 처리한다.
    추가로 m_myEdit 객체가 가지고 있는 함수도 이용할 수 있다.
